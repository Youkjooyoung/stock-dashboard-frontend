# 본인인증 후 기존 회원 자동 감지

> 작성: 2026-04-30
> 대상 저장소: `stock-dashboard-backend`
> 연관 PR: 프론트엔드(`stock-dashboard-react`)에 graceful degradation 형태로 이미 머지됨
> 배포 영향: **다운타임 없음 — 백엔드 응답 필드 추가만으로 즉시 활성화**

---

## 1. 배경 / 문제

회원가입 플로우는 다음 3단계로 구성된다.

1. **본인인증** — PortOne `/api/auth/certify` 호출 → name, phone 반환
2. **정보입력** — 이메일·비밀번호·주민번호·주소 등 입력
3. **이메일 인증** — 가입 완료 후 인증 메일 링크 클릭

문제: 1단계 본인인증 통과 후 정보입력(2단계)에서 **이미 가입된 사용자가 다시 회원가입**을 시도해도 차단되지 않는다. 이메일 중복 검사는 2단계 안에서만 동작하고, 다른 이메일을 입력하면 **동일인이 두 계정을 만들 수 있는 위험**이 있다. 사용자도 인증 메일을 받기 전엔 자신이 이미 가입돼 있다는 사실을 모른다.

본인인증된 name + phone 조합으로 이미 활성 회원이 있다면 **1단계 직후** 즉시 안내하고 LoginPage로 유도해야 한다.

---

## 2. 변경 요약 (TL;DR)

`POST /api/auth/certify`의 응답에 다음 3개 필드를 **추가**한다 (기존 필드는 그대로 유지).

| 필드 | 타입 | 의미 |
|---|---|---|
| `existingMember` | `boolean` | 본인인증된 name+phone 조합으로 활성 회원이 존재하는지 |
| `existingProvider` | `'EMAIL' \| 'KAKAO' \| 'GOOGLE' \| null` | 가입 방식 (소셜 우선) |
| `existingMaskedEmail` | `string \| null` | 마스킹된 이메일 (표시용). 자동 생성 이메일은 null |

프론트는 이미 `existingMember === true`만 체크해 분기하도록 graceful degradation 구현 완료. 필드가 없거나 false면 기존 흐름대로 진행.

## 3. API 스펙 변경 — `POST /api/auth/certify`

### 3.1 요청 (변경 없음)

```http
POST /api/auth/certify
Content-Type: application/json

{
  "impUid": "imp_123456789012"
}
```

### 3.2 응답 — 신규 필드 3종

```jsonc
{
  "name": "홍길동",
  "phone": "01012345678",

  // ▼ 신규
  "existingMember": true,
  "existingProvider": "KAKAO",
  "existingMaskedEmail": "ho***@kakao.com"
}
```

기존 회원이 없을 때:
```jsonc
{
  "name": "홍길동",
  "phone": "01012345678",
  "existingMember": false,
  "existingProvider": null,
  "existingMaskedEmail": null
}
```

### 3.3 하위 호환성

- 필드 누락 허용: 필드가 없으면 프론트는 자동으로 신규 회원 흐름(`existingMember=false`로 간주)으로 진행
- 응답 HTTP 상태 코드 변경 없음 (`200 OK`)
- 필드 명 변경 시 반드시 본 문서 업데이트 후 프론트 동기화

## 4. 비즈니스 로직 (Service 레이어)

### 4.1 본인인증 후 활성 회원 조회 절차

```
1. PortOne API로 imp_uid 검증 → name, phone 추출 (기존 로직)
2. phone 정규화 (하이픈 제거, '+82' → '0' 변환)
3. USERS 테이블에서 NAME = ? AND PHONE = ? AND DELETED_AT IS NULL 조회
4. 결과가 있으면:
   4-1. USER_SOCIAL 테이블에 매핑이 있는지 확인
   4-2. 매핑 있음 → existingProvider = 'KAKAO' 또는 'GOOGLE' (가장 최근 매핑 우선)
   4-3. 매핑 없음 → existingProvider = 'EMAIL'
   4-4. existingMaskedEmail 생성 (§ 7 참조)
5. 결과 없음 → existingMember = false
```

### 4.2 우선순위 (Provider 결정 규칙)

한 사용자가 이메일 가입 후 카카오/구글을 추가 연동했을 수도 있음. 우선순위:

1. 가장 최근 로그인한 provider (가능하면)
2. USER_SOCIAL 매핑이 있으면 → 해당 provider (CREATED_AT 최신)
3. 없으면 → `'EMAIL'`

→ 우선순위는 사용자 경험 관점에서 "기억하기 쉬운 방식"을 표시하기 위함. 비즈니스 정의가 필요하면 우선순위 변경 가능.

## 5. DB 조회 쿼리 예시

### 5.1 USERS + USER_SOCIAL JOIN

```xml
<!-- src/main/resources/mapper/UserMapper.xml -->
<select id="findActiveMemberByIdentity"
        resultType="com.stock.dashboard.dto.UserIdentityResultDto">
  SELECT
    U.USER_ID,
    U.EMAIL,
    (SELECT S.PROVIDER
       FROM USER_SOCIAL S
      WHERE S.USER_ID = U.USER_ID
      ORDER BY S.CREATED_AT DESC
      LIMIT 1)            AS PROVIDER
  FROM USERS U
  WHERE U.NAME = #{name}
    AND REPLACE(U.PHONE, '-', '') = #{phoneNormalized}
    AND U.DELETED_AT IS NULL
  LIMIT 1
</select>
```

### 5.2 phone 정규화

`PHONE` 컬럼이 하이픈 포함/미포함 혼재할 수 있으므로 항상 정규화 후 비교:

```java
String normalize(String p) {
  return p == null ? null : p.replaceAll("\\D", "");
}
```

### 5.3 DELETED_AT IS NULL 필터링

소프트 삭제(2주 보류) 회원은 활성 회원이 아니므로 제외. 탈퇴 보류 회원이 다시 본인인증으로 들어오면 기존의 `/api/auth/check-deleted` 흐름으로 복구 진행 (해당 분기는 별도 — 본 변경과 무관).

## 6. ENUM Provider 처리

### 6.1 응답 값 매핑

`USER_SOCIAL.PROVIDER` 컬럼 값 → 응답 `existingProvider`:

| DB 값 | 응답 값 |
|---|---|
| `'KAKAO'` | `'KAKAO'` |
| `'GOOGLE'` | `'GOOGLE'` |
| (USER_SOCIAL 매핑 없음) | `'EMAIL'` |
| (활성 회원 없음) | `null` |

### 6.2 확장성

향후 `'NAVER'`, `'APPLE'` 등 추가 시 프론트의 `PROVIDER_LABELS` / `PROVIDER_ICONS`에도 반드시 매핑 추가 (`src/components/ExistingAccountStep.jsx`).

## 7. maskedEmail 생성 규칙

### 7.1 일반 이메일

`hong.gildong@naver.com` → `ho***@naver.com`
- 앞 2자 + `***` + 도메인부 (`@` 포함)
- 로컬파트가 1자면: `h***@domain`

### 7.2 소셜 자동생성 이메일은 null

소셜 로그인 시 자동 생성되는 가짜 이메일(`kakao_123456@stock-dashboard.local`, `google_abcd@...` 등)은 사용자가 모르는 식별자이므로 표시하지 않는다.

```java
private boolean isAutoGeneratedEmail(String email) {
  return email != null && (email.startsWith("kakao_") || email.startsWith("google_"));
}
```

→ 자동 생성이면 `existingMaskedEmail = null` 반환. 프론트는 null이면 표시 영역을 숨긴다.

## 9. 보안 고려사항

### 9.1 본인인증된 impUid 필수

`existingMember=true` 반환은 본인인증된 휴대폰 번호로만 트리거되어야 함. **PortOne 검증을 거치지 않고** name+phone만으로 조회 API를 외부에 노출하면 회원 존재 여부가 유출되어 enumeration attack 가능. → 본 변경은 기존 `/api/auth/certify` 응답에만 추가, 별도 엔드포인트 신설하지 않음.

### 9.2 PII 노출 금지

응답에 절대 포함하지 말 것:
- 실제 이메일 (마스킹 후만)
- USER_ID
- 휴대폰 번호 마스킹 (단, 응답에선 phone을 마스킹하지 않음 — 본인인증 결과이므로 본인 정보 OK. 단 프론트 화면 표시용으론 마스킹)
- 가입일/주소/주민번호 등

## 10. 테스트 시나리오

| # | 입력 | 기대 응답 |
|---|---|---|
| 1 | 이메일로 가입한 활성 회원 | `existingMember=true, existingProvider='EMAIL', existingMaskedEmail='ho***@naver.com'` |
| 2 | 카카오로 가입한 활성 회원 | `existingMember=true, existingProvider='KAKAO', existingMaskedEmail=null` |
| 3 | 카카오 가입 후 이메일 추가 연동 | `existingMember=true, existingProvider='KAKAO', existingMaskedEmail='...'` (가장 최근 매핑 우선) |
| 4 | 신규 (미가입) | `existingMember=false, existingProvider=null, existingMaskedEmail=null` |
| 5 | 탈퇴 보류 (DELETED_AT NOT NULL) | `existingMember=false` (활성 아님 — 별도 복구 플로우) |
| 6 | DB 조회 실패 | 500 + `{message:"서버 오류가 발생했습니다."}` (기존 GlobalExceptionHandler 흐름) |

## 11. 배포 순서 (다운타임 없음)

### 11.1 백엔드 먼저 배포 — 응답 필드만 추가

기존 클라이언트 영향 0:
- 기존 프론트는 `existingMember`를 체크하지 않으므로 추가된 3개 필드를 무시 (JSON 추가 필드는 호환)
- 신규 프론트는 이미 `data.existingMember === true`만 분기 트리거, 누락/false면 기존 흐름

### 11.2 프론트는 이미 graceful degradation 상태

→ 프론트 재배포 없이도 백엔드 배포만으로 자동 활성화됨.

배포 순서:
```
1. 백엔드: ./mvnw clean package -DskipTests + sudo systemctl restart stock-dashboard
2. 동작 확인 (curl /api/auth/certify 응답에 신규 필드 노출 확인)
3. 운영 환경 회원가입 1회 직접 테스트 (기존 가입자 휴대폰으로)
```

---

## 12. 롤백 절차

1. 응답 필드를 빈 값으로 두거나 (`existingMember: false` 강제 반환) → 프론트는 자동으로 기존 흐름 복귀
2. 또는 git revert 후 백엔드 재배포

프론트 롤백은 불필요 (기존 흐름 자동 진입).

## 13. 후속 작업 (선택)

- USER_SOCIAL에 `LAST_LOGIN_AT` 컬럼 추가 시 우선순위 결정 정확도 ↑
- 활성 회원이지만 이메일 미인증 상태(`EMAIL_VERIFIED='N'`)면 별도 메시지 표시 검토
- 본인인증 통과 후 7일 이내 가입 미완료 시 재인증 강제 (별도 이슈)
